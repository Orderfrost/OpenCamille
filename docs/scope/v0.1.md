# OpenCamille v0.1 Scope

## Goal

Build a minimal but complete TypeScript Agent Harness that proves OpenCamille can
run an agent loop, call tools safely, manage context, persist traces, load
extensible capabilities, and support future production-grade agent workflows.

v0.1 is not a Claude Code clone and not a vertical product. It is the first
working slice of the harness.

## Product Positioning

OpenCamille is a transparent, controllable, extensible Agent Harness.

It should demonstrate that the project owner can build the core runtime
mechanisms behind practical agents:

- session and lifecycle management
- provider abstraction
- tool execution
- permissions
- MCP integration
- skills and plugins
- memory
- tracing and replay
- verification-driven execution

Reference scenarios can exist, but they are demos for the harness, not the
primary product boundary.

## In Scope

### Core Runtime

- CLI session entry point.
- Single-session agent execution.
- ReAct-style agent loop.
- Message history ownership and persistence.
- Context assembly from system prompt, tools, messages, rules, and memory.
- JSONL trace for model calls, tool calls, permission decisions, and results.

### Provider

- `ProviderAdapter` abstraction.
- One real provider implementation.
- Fake provider for tests.
- Provider-neutral internal message and tool-call types.

v0.1 must keep the adapter boundary clean, but it does not need full feature
parity across multiple providers.

### Tools

- `ToolRegistry`.
- Runtime input validation with `zod`.
- Minimal built-in tools:
  - read file
  - search files
  - edit/write file
  - shell command execution

### Permissions

- Deny-first permission engine.
- Tool categories: safe, write, dangerous.
- Approval flow for write and dangerous operations.
- Session-scoped approval memory.

### MCP

- Minimal MCP client support.
- At least one working local MCP integration path.
- Treat MCP tools as normal registered tools after adaptation.

Full ecosystem compatibility is not required in v0.1.

### Plugins

- Minimal local plugin loading.
- Plugin manifest.
- Plugin can contribute at least one of:
  - tool
  - skill
  - command
  - rule

No marketplace, remote install, version solver, or plugin sandbox in v0.1.

### Skills

- Minimal skill format.
- Skill can inject prompt instructions and optional tool allowlist.
- Skill loading/unloading is explicit.

No complex multi-layer skill package format in v0.1.

### Memory

- Minimal long-term memory.
- Human-readable local storage.
- Read relevant memories into context.
- Write explicit user-approved facts or project conventions.

No vector database, semantic search, or automatic large-scale memory extraction
in v0.1.

### Verification

- Run a verification command through the tool system.
- Feed failures back into the agent loop.
- Record verification attempts in trace.

### Testing

- Unit tests for:
  - permission rules
  - tool schema validation
  - provider adapter boundary
  - session state transitions
  - trace serialization

## Lower Priority In v0.1

- Multi-agent orchestration.
- Web UI.
- Advanced trace visualization.
- Full second provider support.
- Compaction agent.

These are framework-relevant, but they should not block the first complete
runtime slice.

## Out Of Scope

- IDE integration.
- Plugin marketplace.
- Cloud execution.
- Team collaboration features.
- Full autonomous background daemon.
- Enterprise policy management.

## Success Criteria

v0.1 is complete when:

1. A user can start a CLI session and ask the agent to perform a small task.
2. The agent can call registered tools through validated schemas.
3. Write and shell actions pass through permission checks.
4. The session records a replayable JSONL trace.
5. A skill can be loaded and change agent behavior.
6. A local plugin can be discovered and loaded.
7. A minimal MCP tool can be adapted into the tool registry.
8. Long-term memory can be read into context and updated deliberately.
9. A verification command can be run and its result can guide the next loop.
10. Core behavior is covered by tests.

## Demo Target

The default demo should be small and local:

1. Load a repo.
2. Ask OpenCamille to inspect it.
3. Load a skill.
4. Call file/search/shell tools.
5. Request permission before changing files or running risky commands.
6. Run a verification command.
7. Produce a trace that explains what happened.

The demo should prove the harness, not compete with mature coding agents.
