Agent Harness / 已确认草案

OpenCamille 新版完整架构图

当前架构把 OpenCamille 定位为 Agent Harness：负责运行时、上下文、工具、权限、生命周期、记录与扩展边界。 目标不是复刻某个垂直助手，而是证明一个可控、可扩展、可审计的 Agent Runtime 能从底层跑通。

03

架构原则与取舍

最新文档把“为什么这样设计”写清楚了：先证明 Harness 底座，避免空壳模块，把状态、事件、权限和持久化边界固定下来。

P1

状态写入集中

live Session 和 ConversationHistory 只能由 Runtime Control 写入，避免 CLI、Tools、Recorder、AgentLoop 分散改状态。

P2

执行循环收敛

Agent Runtime 只保留 AgentLoop 和 ContextManager。ToolExecutor、ExecutionStrategy 等等到真实复杂度出现后再拆。

P3

能力放 Services

Tools、Skills、Memory、MCP 是 Agent 能力；Lifecycle、Hooks、Plugins、Recorder 是运行时服务。

P4

底层边界放 Infrastructure

Provider、Config、Permission、MCP transport、命令执行、路径边界放在底层，隔离外部系统差异和副作用风险。

P5

事件用于观察

EventBus 服务 streaming、recording、hooks 和 replay timeline，但不替代状态源；resume 只读 checkpoint。

P6

v0.1 不做空壳框架

少文件、少抽象、职责清晰、边界强约束。每个非平凡逻辑都要留下最小可运行验证。

04

关键契约

这些是 v0.1 不能打破的架构边界，也是面试展示时最能说明工程判断的部分。

Approval resume

Tools 返回 approval_required，AgentLoop 交回控制权，Runtime Control 暂停并恢复同一个 tool call。

Tools -> AgentLoop -> Runtime Control -> Surface

Provider streaming

ProviderAdapter 接收厂商无关 ContextAssembly，输出厂商无关 stream items，SDK 形状不泄漏到 Agent Runtime。

ContextAssembly -> ProviderStreamItem

Tool result hygiene

工具输出在进入模型上下文、Transcript 或 Events 之前统一 normalize、truncate、redact。

validation / permission / redaction

Context overflow

ContextManager 报 overflow；AgentLoop 只调用 Memory.compact() 一次并重试一次，不静默截断。

overflow -> compact once -> rebuild once

Recorder semantics

events.jsonl 是机器读 timeline truth；transcript.jsonl 是人读会话记录；checkpoint.json 是 resume source。

.opencamille/runs/<runId>/

Secrets boundary

Config 有确定优先级，secret 值不能写入事件、转录、checkpoint 或工具记录。

defaults -> env -> user -> project -> CLI

05

模块责任矩阵

这一段把五层图拆成可实现模块。重点不是文件数量，而是谁拥有状态、谁只返回结果、谁负责安全边界。

Surface

CLI

输入
用户文本、命令、flags、审批选择
存在原因
把交互方式和 Agent 运行逻辑拆开，避免后续 Web/API 重复业务逻辑
输出
流式事件展示、审批提示、最终消息
禁止
不调用 provider、不执行 tool、不写 Recorder

Runtime Control

Session

拥有
sessionId、runId、status、conversationHistory、pendingApproval、budget、sessionSummary
存在原因
把 live state 的写入权收敛到一个地方，保证 approval、resume、debug 可控
状态
idle / running / waiting_approval / paused / finished / failed / cancelled
规则
v0.1 不拆 RunState、ApprovalState、BudgetState

Runtime Control

CommandDispatcher

职责
接收 Surface 命令，决定是用户消息、控制命令还是恢复审批
位置
属于 Runtime Control，不放进 Agent Services
写入
只通过 Runtime Control 追加 ConversationHistory
边界
不做模型调用，不做工具执行

Agent Runtime

AgentLoop

循环
build context -> stream model -> collect tool_use/final message -> run tool or finish
存在原因
Agent Harness 的核心循环必须集中，否则 streaming、tool use、approval、compaction 会分散
守卫
max turns、budget、abort signal inline 检查
事件
只发 turn/model/tool/compaction 生命周期事件

Agent Runtime

ContextManager

读取
ConversationHistory、Memory、active skill、tool schemas、MCP metadata
存在原因
统一组装厂商无关上下文，避免 provider 细节和记忆/技能注入散落到各层
输出
厂商无关 ContextAssembly: system / tools / messages
溢出
返回 context_overflow，不在内部静默截断

Agent Services

Tools

入口
register、list schemas、resolve、runToolCall
存在原因
统一内置 tools、Skill tool、MCP tools 的 schema、解析、权限和执行入口
路径
zod validation -> PermissionEngine -> callable -> normalize/truncate/redact
返回
ok / error / approval_required

Agent Services

Skills

格式

存在原因
把任务知识包渐进式加载进上下文，避免所有任务说明常驻 base prompt
机制
SkillIndex 记录元数据，built-in Skill tool 激活 body
暂缓
dynamic command injection、forked subagents、live watching

Agent Services

MCP

v0.1
stdio JSON-RPC、listTools、callTool
存在原因
遵循官方 MCP 接入外部工具生态，同时让外部工具走统一权限与记录路径
集成
MCP tools 适配进 Tools，共用 validation、permission、recording path
不做
HTTP transport、auth、resources、prompts、sampling、hot reload

Runtime Services

Lifecycle / Hooks

Lifecycle
事件名和 in-process EventBus 的唯一来源
存在原因
为 streaming UI、Recorder、Hooks、Replay、外部 Eval 提供统一时间线
Hooks
固定 hook points；能阻塞/修改的 hook 不等同于 EventBus subscriber
Emitter

Runtime Services

Recorder

存在原因

transcript.jsonl

checkpoint.json
resume source；v0.1 不用 event sourcing 恢复

ProviderAdapter

输入

存在原因
隔离不同 LLM API 的 system/tools/messages、stream、tool_use、usage、error 差异
输出
model_delta / tool_use / final_message / usage / error

这里展示实现时要稳定下来的协议形状。面试时可以用它说明：架构不是画层，而是定义跨模块的合同。

Recorder

事件用于 replay/eval/debug；转录用于展示；checkpoint 用于 resume，不靠事件重放恢复。

Config precedence

built-in defaults
env
user config
project config
CLI flags

优先级确定；secret 值不能进入 events、transcript、checkpoint 或 tool/model records。

不要事件化内部 debug 细节

Tools.resolve 查了几个 registry、ContextManager 估算了多少 tokens、PermissionEngine 匹配了哪条内部 rule、 ProviderAdapter 内部 retry 细节都可以作为 debug log，但不进入稳定 RunEvent。

Approval path

  1. AgentLoop 收到 tool_use，发 tool_requested。
  2. Tools.runToolCall 做 zod 校验，并请求 PermissionEngine.authorize。
  3. PermissionEngine 返回 ask。
  4. Tools 返回 approval_required。
  5. AgentLoop 把 pending approval 返回 Runtime Control。
  6. Runtime Control 设置 status = waiting_approval。
  7. Surface 展示审批；批准后 Runtime Control 恢复同一个 tool call。

Context overflow path

  1. ContextManager 检测上下文过大，返回 context_overflow。
  2. AgentLoop 发 context_overflow / compaction_started。
  3. Memory.compact() 更新 SessionSummary。
  4. AgentLoop 只 rebuild 一次。
  5. 仍失败则返回错误，不静默丢弃消息。

Session / Run

session_created · session_resumed · session_closed · run_started · run_finished · run_failed · run_cancelled

Turn / Context

turn_started · context_built · context_overflow · turn_finished · turn_failed

Model / Tool

model_started · model_delta · model_finished · model_failed · tool_requested · tool_started · tool_finished · tool_failed · tool_denied

Approval / Recovery

approval_requested · approval_resolved · compaction_started · compaction_finished · compaction_failed · checkpoint_started · checkpoint_finished · checkpoint_failed

Hook point v0.1 行为 原因
before_tool_call 可 deny / modify args 工具调用是明确副作用边界，需要拦截能力。
after_tool_call 可 modify result 结果进入模型和记录前可做规整、脱敏或展示适配。
before_user_message 可 deny / modify text 用户输入进入 ConversationHistory 前是最后治理点。
before_message_display 只修改 display 展示适配不能反写 canonical conversation state。
其他 hook points 只观察 v0.1 放置扩展点，但不提前实现完整插件执行模型。
07

v0.1 最小切片

v0.1 证明 Harness 能跑通，不做成熟编码 Agent 的全部能力。

领域 v0.1 包含 明确不做
Surface CLI，streaming 输出，approval prompt，基本 command 输入 Web / API / IDE
Runtime 单 live Session，ConversationHistory 唯一写入，CommandDispatcher，checkpoint resume 多 Session 调度、复杂 state class 拆分
Agent ReAct-style AgentLoop，ContextManager，ProviderAdapter streaming contract，overflow compact once ToolExecutor、ExecutionStrategy、TerminationGuard、StateMachineEngine
Services Tools、Skills、SessionSummary、PersistentMemory、stdio MCP tools、Lifecycle、Hooks、Recorder Subagents、RAG/vector memory、完整插件包格式、MCP HTTP/auth/resources/prompts
Infrastructure ProviderAdapter、Config、PermissionEngine、MCPClient、CommandRunner、WorkspacePath 真实 sandbox、Storage abstraction、ProviderFactory、PolicyStore、SecretsManager、NetworkClient

v0.1 明确不做

Web UI · API surface · IDE integration · Subagents · RAG/vector memory · plugin marketplace · Skill fork subagent · dynamic command injection · live file watching · MCP OAuth/auth/sampling/roots/elicitation · OpenTelemetry export · SQLite/remote storage · multi-provider parity

08

实现路径与验收

最新文档更强调实现前读取顺序、最小 Harness 闭环和验收条件，而不是堆模块。

必读顺序

docs/README.md→ architecture.md→ scope/v0.1.md→ spec/v0.1-architecture.md→ ai-development-guide.md→ 对应 architecture/*.md

v0.1 Harness 闭环

  1. 启动 CLI session。
  2. 打开一个小 repo，让 Agent inspect repo。
  3. 通过 ProviderAdapter 看到 streaming model response。
  4. 通过 built-in Skill tool 激活一个 Skill。
  5. 读取、搜索、编辑文件。
  6. 写入或 shell 前请求 approval，并恢复同一个 tool call。
  7. 调用一个 stdio MCP tool。
  8. 生成可检查的 events、transcript、checkpoint。

实现前必须回答

  1. 当前任务属于哪个模块？
  2. 该模块为什么存在，唯一职责是什么？
  3. 它依赖谁，谁依赖它？
  4. 它是否会写 live Session 或 Recorder？
  5. 它是否可能触发 permission ask 或接触 secrets？
  6. 它是否属于 v0.1 scope？
  7. 如何测试，是否影响架构边界？
.opencamille/runs/<runId>/
├── events.jsonl 机器读 timeline truth / eval / replay / debug
├── transcript.jsonl 人读会话记录 / UI / export / demo
└── checkpoint.json resume source / rehydrate live Session
