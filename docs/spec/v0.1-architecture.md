# OpenCamille v0.1 架构实现规范

> 更新日期：2026-06-29  
> 作用：给开发者和 AI Agent 的 v0.1 直接实现依据。

## 1. 实现原则

v0.1 要实现最小完整闭环，不实现空壳框架。

原则：

```text
少文件
少抽象
职责清晰
边界强约束
每个非平凡逻辑有最小测试
```

建议目录：

```text
src/surfaces/cli.ts

src/runtime-control/session.ts
src/runtime-control/commands.ts

src/agent-runtime/loop.ts
src/agent-runtime/context.ts

src/services/tools.ts
src/services/skills.ts
src/services/memory.ts
src/services/mcp.ts
src/services/lifecycle.ts
src/services/hooks.ts
src/services/recorder.ts

src/infrastructure/provider.ts
src/config.ts
src/infrastructure/permissions.ts
src/infrastructure/mcp.ts
src/infrastructure/command.ts
src/infrastructure/workspace.ts
```

这是建议，不是强制。实现很小时可以合并文件，但不能新增未确认模块。

## 2. Runtime Types

这些是 v0.1 的概念类型。实现时可以拆分文件，但契约不能随意改变。

```ts
type SessionStatus =
  | "idle"
  | "running"
  | "waiting_approval"
  | "paused"
  | "finished"
  | "failed"
  | "cancelled";

type MessageRole = "user" | "assistant" | "tool" | "system";

type Message = {
  id: string;
  role: MessageRole;
  content: string;
  toolCallId?: string;
  toolCalls?: ToolCall[];
  createdAt: string;
};

type Session = {
  sessionId: string;
  runId: string;
  status: SessionStatus;
  conversationHistory: Message[];
  pendingApproval?: ApprovalRequest;
  budget: {
    maxTurns: number;
    turnCount: number;
  };
  sessionSummary?: string;
};

type ContextAssembly = {
  system: string;
  tools: ToolDefinition[];
  messages: Message[];
};

type ContextSourceKind =
  | "builtin"
  | "config"
  | "file"
  | "session"
  | "tools"
  | "mcp"
  | "skills"
  | "memory";

type ContextTarget = "system" | "tools" | "messages";

type ContextSource = {
  id: string;
  kind: ContextSourceKind;
  target: ContextTarget;
  priority: number;
  content?: string;
  tools?: ToolDefinition[];
  messages?: Message[];
  fingerprint: string;
};

type ContextBudget = {
  maxChars: number;
  usedChars: number;
  bySource: Array<{
    sourceId: string;
    chars: number;
  }>;
  overflow: boolean;
};

type ContextSnapshot = {
  sessionId: string;
  version: number;
  sources: Array<{
    id: string;
    kind: ContextSourceKind;
    target: ContextTarget;
    fingerprint: string;
    chars: number;
  }>;
  context: ContextAssembly;
  budget: ContextBudget;
  createdAt: string;
};

type ContextBuildInput = {
  session: Session;
  config: unknown; // 实现中使用 Config
  extraSources?: ContextSource[];
  ruleFiles?: string[];
  maxChars?: number;
};

type ContextBuildResult =
  | {
      status: "ok";
      context: ContextAssembly;
      snapshot: ContextSnapshot;
    }
  | {
      status: "context_overflow";
      reason: string;
      snapshot: ContextSnapshot;
    };

type ProviderStreamItem =
  | { type: "model_delta"; text: string }
  | { type: "tool_use"; toolCall: ToolCall }
  | { type: "final_message"; message: Message }
  | { type: "usage"; inputTokens?: number; outputTokens?: number }
  | { type: "error"; error: string };

type ToolDefinition = {
  name: string;
  description: string;
  inputSchema: unknown; // 实现中使用 zod schema
  category: "safe" | "write" | "dangerous";
};

type ToolCall = {
  id: string;
  name: string;
  input: unknown;
};

type ToolResult =
  | {
      status: "ok";
      toolCallId: string;
      content: string;
      recordContent?: string;
      truncated: boolean;
    }
  | {
      status: "error";
      toolCallId: string;
      error: string;
    }
  | {
      status: "approval_required";
      request: ApprovalRequest;
    };

type PermissionDecision = "allow" | "ask" | "deny";

type ApprovalRequest = {
  id: string;
  toolCall: ToolCall;
  reason: string;
  createdAt: string;
};

type RunEvent = {
  id: string;
  runId: string;
  sessionId: string;
  turnId?: string;
  type: string;
  timestamp: string;
  data?: Record<string, unknown>;
};

type TranscriptItem = {
  id: string;
  runId: string;
  sessionId: string;
  turnId?: string;
  role: "user" | "assistant" | "tool" | "system";
  content: string;
  createdAt: string;
};

type Checkpoint = {
  runId: string;
  sessionId: string;
  status: SessionStatus;
  conversationHistory: Message[];
  pendingApproval?: ApprovalRequest;
  budget: Session["budget"];
  sessionSummary?: string;
  contextSnapshot?: ContextSnapshot;
  createdAt: string;
};
```

实现说明：

- `ToolDefinition.inputSchema` 必须在代码中使用 zod。
- `RunEvent.type` 可以先用 string + 常量表，稳定后再收窄为 union。
- `ProviderStreamItem` 是真实协议 adapter 和 mock fetch 单元测试共同验证的归一化契约。
- 真实 API 用于集成验证和 demo；单元测试用 mock fetch 固定 SSE 输入，避免依赖外部网络。
- `ContextBuildInput.config` 在实现中使用 `Config` 类型；这里用 `unknown` 避免规范片段和具体文件导入耦合。
- `Checkpoint.contextSnapshot` 必须是已脱敏 snapshot，不能包含 provider secrets。

## 3. Session 与 Runtime Control

v0.1 使用一个 Session 对象，不拆多个 state class。

Runtime Control 必须保证：

```text
ConversationHistory 只有 Runtime Control 写
AgentLoop 返回结果，不直接写 Session
Tools 返回结果，不直接写 Session
Recorder 写 checkpoint，不拥有 live Session
```

Approval resume：

```text
Tools.runToolCall()
  -> PermissionEngine returns ask
  -> approval_required
  -> AgentLoop returns pending approval
  -> Runtime Control status = waiting_approval
  -> CLI asks user
  -> Runtime Control resumes same tool call
```

测试要求：

```text
approval_required 会保存 pendingApproval
用户 approve 后恢复同一个 toolCall.id
用户 deny 后不会执行 tool
```

## 4. ProviderAdapter

ProviderAdapter 隐藏厂商 API 差异。

输入：

```text
ContextAssembly
```

输出：

```text
ProviderStreamItem
  model_delta
  tool_use
  final_message
  usage
  error
```

Provider-specific 结构不能泄漏到 AgentLoop。

当前实现基线：

```text
Config 解析 active Provider Profile
ProviderAdapter 只根据 provider.protocol 选择协议 adapter
vendor/name 不参与 adapter 选择
使用 fetch 直接调用 HTTP/SSE，不引入厂商 SDK
```

支持的协议值只包括：

```text
openai-responses
openai-chat-completions
anthropic-messages
```

开发初期不兼容旧别名，例如 `openai`、`anthropic`。OpenAI-compatible 厂商通过 Provider Profile 声明 `protocol: openai-chat-completions` 复用同一个 adapter，不新增每厂商 adapter。

当前不做：

```text
ProviderFactory 独立抽象
SecretsManager
厂商 SDK
每厂商 adapter
根据 vendor 推断协议
```

## 5. ContextManager

ContextManager 负责构建厂商无关的 ContextAssembly，并维护 context budget、ContextSnapshot 缓存和 checkpoint 恢复协作。

v0.1 的输入按来源组织，而不是按未来模块字段不断扩展：

```text
ContextSource:
  builtin
  config
  file
  session
  tools
  mcp
  skills
  memory
```

输出：

```text
ContextAssembly
  system
  tools
  messages
```

如果上下文超限：

```text
return context_overflow
```

不要在 ContextManager 里偷偷截断 ConversationHistory。overflow 的处理由 AgentLoop 触发 Memory.compact()。

ContextManager 可以自己读取少量固定规则文件，例如 `AGENTS.md` / `Agent.md`；文件不存在或为空时跳过。它不扫描 MCP、不扫描 Skill、不检索 Memory、不执行 Tool。

详细开发依据见：

```text
docs/implementation/v0.1/context-manager-design.md
```

## 6. AgentLoop

v0.1 执行流程：

```text
while not done:
  result = ContextManager.build({ session, config, extraSources })

  if result.status == "context_overflow":
    emit context_overflow
    emit compaction_started
    Memory.compact()
    emit compaction_finished
    rebuild context once

  emit model_started
  stream = ProviderAdapter.stream(result.context)
  collect model_delta / tool_use / final_message / usage
  emit model_delta while streaming
  emit model_finished

  if tool_use:
    emit tool_requested
    emit tool_started
    result = Tools.runToolCall(toolUse)

    if result.status == "approval_required":
      emit approval_requested
      return pending approval to Runtime Control

    emit tool_finished / tool_failed
    return tool result to Runtime Control
    continue

  return final assistant message to Runtime Control
```

AgentLoop 不直接写 Session。

不要实现：

```text
ToolExecutor
ExecutionStrategy
TerminationGuard
AgentRuntimeEvent
StateMachineEngine
```

## 7. Tools

Tools 负责：

```text
register
list schemas
resolve
runToolCall
```

`runToolCall()` 具体流程：

```text
resolve tool
validate input with zod
PermissionEngine.authorize()
if ask -> return approval_required
if deny -> return error/denied
if allow -> execute callable
normalize result
truncate large output
redact secrets
return ok/error
```

Tools 不能：

```text
修改 ConversationHistory
等待用户 approval
写 Recorder 文件
调用 ProviderAdapter
决定 AgentLoop 下一步
```

ToolResult 进入以下位置前必须规范化/脱敏：

```text
model context
events.jsonl
transcript.jsonl
```

## 8. PermissionEngine

PermissionEngine 只做判断。

输入：

```text
action/tool name
tool input summary
category
config rules
session approval memory if provided by Runtime Control
```

输出：

```text
allow | ask | deny
```

它不等待用户，不保存 pendingApproval。

## 9. Lifecycle/EventBus

EventBus 是进程内 pub/sub。

最小接口：

```ts
type EventBus = {
  emit(event: RunEvent): void | Promise<void>;
  subscribe(listener: (event: RunEvent) => void | Promise<void>): () => void;
};
```

只有 AgentLoop 和 Runtime Control 直接 emit。

不要事件化内部 debug 细节，例如：

```text
Tools.resolve 查了几个 registry
ContextManager 估算了多少 tokens
PermissionEngine 匹配了哪条内部 rule
ProviderAdapter 内部 retry 细节
```

这些可以作为 debug log，不进入稳定 RunEvent。

## 10. Hooks

HookPoint 必须先埋好，但 v0.1 行为要克制。

HookPoint：

```text
session_start
session_resume
session_stop
before_user_message
after_user_message
before_context_build
after_context_build
before_model_call
after_model_call
before_tool_call
after_tool_call
before_permission_request
after_permission_resolved
before_compaction
after_compaction
before_checkpoint
after_checkpoint
before_message_display
```

v0.1：

```text
大多数只观察
before_tool_call 可 deny/modify args
after_tool_call 可 modify result
before_user_message 可 deny/modify text
before_message_display 只允许修改展示内容
```

Hooks 不通过 EventBus 阻塞流程。

## 11. Recorder

Recorder 写：

```text
.opencamille/runs/<runId>/
  events.jsonl
  transcript.jsonl
  checkpoint.json
```

写入路径：

```text
EventBus -> Recorder.appendEvent(event)
Runtime Control -> Recorder.appendTranscript(item)
Runtime Control -> Recorder.writeCheckpoint(snapshot)
```

语义：

```text
events.jsonl      eval / replay / debug
transcript.jsonl  readable UI/export/demo
checkpoint.json   resume
```

不要用 events replay 恢复 Session。Resume 只读 checkpoint。

## 12. Skills

必须遵循 Anthropic Agent Skills。

最小结构：

```text
skill-name/
  SKILL.md
```

frontmatter：

```yaml
name: skill-name
description: short trigger-oriented description
```

v0.1：

```text
scan user/project skill dirs
build SkillIndex
register built-in Skill tool
Skill tool activates skill body
ContextManager includes active skill body
```

不做自定义 Skill 协议。

## 13. MCP

v0.1：

```text
Infrastructure/MCPClient
  stdio JSON-RPC
  listTools
  callTool

Services/MCP
  adapt MCP tools into Tools
```

不做：

```text
HTTP transport
auth
resources
prompts
sampling
roots
elicitation
hot reload
```

## 14. Memory

v0.1：

```text
SessionSummary
PersistentMemory
```

Compaction：

```text
ContextManager -> context_overflow
AgentLoop -> Memory.compact()
ContextManager -> rebuild once
```

不做：

```text
WorkingNotes
RetrievedMemory
RAG
vector DB
automatic long-term fact extraction
```

## 15. Config

Config 属于 Infrastructure 层，负责把配置文件、环境变量和 CLI flags 解析成已校验的 Runtime Config 与当前 Provider Profile。

优先级从低到高：

```text
built-in defaults
user config
project config
local config
env
CLI flags
```

要求：

```text
使用 zod validation
Provider Profile 与 Runtime Config 分层
协议只接受 openai-responses / openai-chat-completions / anthropic-messages
厂商只通过 Provider Profile 配置，不根据厂商名推断协议
secrets 只能读
secrets 不写入 events/transcript/checkpoint
tool/model records 中出现已知 secret value 必须 redact
```

Provider Profile 字段只描述 provider 接入：

```text
vendor
protocol
baseUrl
version
model
apiKey
authToken
headers
organization
project
beta
```

Runtime Config 字段描述运行行为，例如 `maxTurns`、`maxOutputTokens`、采样参数、tool choice、permission default、timeout/retry 控制。不要把运行行为放进 Provider Profile。

## 16. Infrastructure 小函数

`runCommand()`：

```text
cwd
timeout
abortSignal
stdout/stderr limit
exit code
```

`resolveWorkspacePath()`：

```text
relative/absolute path handling
workspace root boundary
path traversal protection
symlink risk if needed
```

不要做 `Shell` class 或 `FileSystem` module。

## 18. 实现顺序

推荐顺序：

```text
1. Config + runtime types
2. Lifecycle/EventBus
3. Recorder
4. ProviderAdapter 协议基线与归一化测试
5. Session + Runtime Control
6. ContextManager
7. Tools + PermissionEngine + normalization/redaction
8. AgentLoop
9. CLI Surface
10. 真实协议 adapter 手动验证
11. Skills + built-in Skill tool
12. stdio MCP tools
13. Memory compaction
14. approval resume + checkpoint resume 集成测试
```

不要让一个 AI Agent 一次实现完整 Harness。每次只给一个窄任务。
